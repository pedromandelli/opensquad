# Template Designer — Design Spec

**Date:** 2026-03-27
**Status:** Draft
**Skill name:** `template-designer`

## Summary

A skill that opens an interactive visual companion (local browser) for choosing and refining image templates during squad creation. When the Architect detects a squad needs an image design agent, it offers the user the option to select a visual identity template. The chosen template's HTML and style rules are saved into the design agent's instructions so it can consistently reproduce the visual style.

## Architecture

### Skill Structure

```
skills/template-designer/
  ├── SKILL.md                    ← Skill definition (type: hybrid)
  ├── scripts/
  │   ├── server.js               ← HTTP server (~100 lines, pure Node.js, zero deps)
  │   ├── start-server.sh         ← Start script (handles Windows/background modes)
  │   ├── stop-server.sh          ← Manual shutdown if needed
  │   ├── frame-template.html     ← HTML wrapper (dark theme, base CSS, event JS)
  │   └── helper.js               ← Client-side JS (report clicks/selections)
  └── base-templates/
      ├── model-a.html            ← Base model 1
      ├── model-b.html            ← Base model 2
      └── model-c.html            ← Base model 3
```

### Components

1. **HTTP Server** (`scripts/server.js`) — Pure Node.js, no npm dependencies. Binds to random high port on `127.0.0.1`. Watches `content_dir` and serves the newest HTML file by modification time. Injects base CSS theme and helper JS. Records browser interactions to `state_dir/events.jsonl`. Auto-terminates after 30 minutes of inactivity. Writes `server-info.json` on start and `server-stopped` on exit.

2. **Base Templates** (`base-templates/`) — 3 distinct HTML/CSS models at 1080x1440 (carousel format). These serve as starting points for adaptation, never shown directly to the user. Specific styles to be approved by the user before inclusion. At runtime, the agent adapts them to the squad's context.

3. **SKILL.md** — Type `hybrid`. Contains instructions for when to invoke, how to operate the server, how to adapt templates, how to extract style rules, and how to save outputs.

### Session Directory

```
squads/{code}/_build/template-session/
  ├── content/              ← HTMLs generated by the agent
  ├── state/
  │   ├── server-info.json  ← Port, URL, paths
  │   ├── events.jsonl      ← User browser interactions
  │   └── server-stopped    ← Written on shutdown
```

Lives inside `_build/` as a temporary creation artifact, alongside `discovery.yaml` and `design.yaml`.

## Flow

### During Squad Creation (between Design and Build phases)

```
Architect (Design phase output) → design.yaml includes image design agent
  → Architect asks: "Squad includes an image design agent.
     Want to choose a visual template now? (You can do this later too.)"
    → Yes → invoke template-designer skill
              → start server → generate adapted templates → show in browser
              → user picks + refines via terminal text → approves
              → save HTML ref + style specs into design.yaml context
              → server dies by inactivity timeout
    → No  → Build phase continues without fixed template
              → user can invoke later via natural language
```

### Outside Creation Flow (later editing)

User asks in natural language: "edit the designer template", "change visual identity of squad X", etc. The Architect:
1. Identifies the squad and its design agent
2. Invokes the `template-designer` skill
3. If `template-reference.html` and `visual-identity.md` already exist, loads them as starting point (user refines from existing rather than starting from scratch)

## Template Adaptation Process

1. Agent reads the 3 base models from `skills/template-designer/base-templates/`
2. Reads squad context: `discovery.yaml` (platform, domain, tone), `design.yaml` (agents, purpose), and Sherlock data if available (color palette, visual patterns from reference profiles)
3. Generates 3+ adapted HTML variations — adjusting colors, typography, layout, domain-relevant example content, and resizing to target viewport if different from 1080x1440
4. Writes adapted HTMLs to the session's `content_dir`
5. Shows in browser for user selection

### Iteration Loop

- User gives text feedback in terminal ("darker background", "cleaner typography", "mix template B layout with template A colors")
- Agent generates new version as new HTML file (never overwrites — `template-v1.html`, `template-v2.html`, etc.)
- Repeats until user approves

## Output: What Gets Saved

When the user approves the final template, two artifacts are created:

### A) HTML/CSS Reference

**Path:** `squads/{code}/pipeline/data/template-reference.html`

The complete HTML/CSS of the approved template. The design agent uses this as a literal example when producing images.

### B) Structured Style Rules

**Path:** `squads/{code}/pipeline/data/visual-identity.md`

Extracted from the approved template:
- **Color palette** — primary, secondary, background, text, accents (hex codes)
- **Typography** — families, weights, minimum sizes per element
- **Layout** — spacing, margins, alignment, grid rules
- **Composition rules** — logo placement, image treatment, visual hierarchy
- **Target viewport** — dimensions and adaptation rules

### Integration with Design Agent

- `squad.yaml` gains both files in its `data:` field
- The design agent's `.agent.md` references both files in its instructions
- The design agent's tasks include the rule: "always follow `visual-identity.md` and use `template-reference.html` as the base model"

This way the design agent can both faithfully reproduce the template (knows the "how" from HTML) and adapt intelligently (knows the "why" from the rules).

## Server Specification

- **Runtime:** Pure Node.js, zero npm dependencies
- **Bind:** `127.0.0.1`, random high port
- **Content serving:** Watches `content_dir`, serves newest file by mtime
- **Frame injection:** Wraps content fragments in `frame-template.html` (dark theme, base CSS, selection indicators). Full HTML documents (`<!DOCTYPE` or `<html>`) served as-is with only helper.js injected.
- **Events:** Browser clicks/selections recorded to `state_dir/events.jsonl` as JSON lines
- **Timeout:** 30 minutes of inactivity (no HTTP requests) → auto-shutdown
- **Shutdown signal:** Writes `server-stopped` to `state_dir`
- **Startup output:** Writes `server-info.json` to `state_dir` with `{ port, url, content_dir, state_dir }`
- **Platform handling:** `start-server.sh` detects Windows/Git Bash and switches to foreground mode automatically. On Windows, the SKILL.md instructs the agent to use `run_in_background: true` on the Bash tool call so the server process survives across conversation turns. The agent then reads `server-info.json` on the next turn to get the URL.

## Architect Integration

- The `architect.agent.yaml` gains awareness of the `template-designer` skill in its skill list
- The Design phase prompt (`design.prompt.md`) includes logic to check if the squad has an image design agent and offer template selection
- The trigger is the presence of `image-creator` (or similar) in the squad's skill list within `design.yaml`
- The user can also trigger it later via natural language — the Architect recognizes intent and invokes the skill on the relevant squad

## Constraints

- No dependency on superpowers plugin or any external plugin
- Server is pure Node.js with zero npm dependencies
- Base templates must be approved by the user before inclusion in the skill
- Base template format is 1080x1440 (carousel), adapted at runtime to target viewport
- User interaction is text-only in terminal; browser is display-only (click to select, but no editing)
